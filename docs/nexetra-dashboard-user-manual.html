Nexetra Media Dashboard

User Manual  ·  Version 1.0  ·  2026-06-09

Overview

The Nexetra Media Compute Dashboard is a real-time web application that gives you a single-pane view of your distributed AI video production pipeline.

  • Node health — every eligible compute host, capabilities, and active job status
  • Batch jobs — per-job progress through the 5-stage media pipeline with AI goal descriptions
  • Event history — timestamped log of every stage start, pass, fail, and completion
  • Pool statistics — remote-ready, reachable, Ollama nodes, active leases, jobs defined

The dashboard auto-updates every 4 seconds via Server-Sent Events (SSE). No manual refresh needed. Protected hosts (linux-1, linux-2, das-Mac-mini.local, sql-server) are never shown.

Access the Dashboard

URL http://10.0.0.200:7800

Accessible from any machine on the lab network. No authentication required — it is an internal read-only monitoring tool.

Header Bar

The top bar shows:

  • Application name and logo
  • A LIVE badge with a green pulsing dot confirming the SSE connection is active
  • An "Updated HH:MM:SS" timestamp showing when data was last refreshed

If the SSE connection drops, the badge changes to "⚠ Reconnecting…" and automatically reconnects.

Event History

A scrollable table showing the last 60 events from all batch runs (most-recent first):

ColumnContents
TimeLocal time the event was recorded
JobJob ID (path and extension stripped)
StagePipeline stage name, or blank for batch-level events
Eventbatch_start, stage_start, stage_pass, stage_fail, batch_done, batch_fail
DetailHost name, host list, or error context (monospaced)

Events are colour-coded: blue = start, green = pass/done, red = fail, dim = release.

History is stored in output/job_runs.jsonl and persists across service restarts.

Running a Batch Job

Standard run

cd nexetra-media
python pipeline\run_batch_pool.py --job jobs\what-is-nexetra-live-es.json

The runner will:

  1. Run a health check across all eligible hosts
  2. Acquire leases on remote-ready hosts only
  3. Execute all 5 stages on the anchor host
  4. Log every stage event to output/job_runs.jsonl
  5. Release all leases when done (pass or fail)

Dry run (no LLM/TTS calls)

python pipeline\run_batch_pool.py --job jobs\what-is-nexetra-live-es.json --dry-run

Local fallback (if remote hosts unavailable)

python pipeline\run_batch_pool.py --job jobs\what-is-nexetra-live-es.json --allow-local-fallback

Health check only

python pipeline\run_batch_pool.py --job jobs\what-is-nexetra-live-es.json --health-only
The dashboard updates automatically as events are written. There is no need to reload the page.

Running Individual Stages

Each pipeline stage can be invoked directly:

python pipeline\scriptgen\agent.py --job jobs\what-is-nexetra-live-es.json
python pipeline\translate\agent.py --job jobs\what-is-nexetra-live-es.json
python pipeline\tts\agent.py       --job jobs\what-is-nexetra-live-es.json
python pipeline\assembly\agent.py  --job jobs\what-is-nexetra-live-es.json
python pipeline\export\agent.py    --job jobs\what-is-nexetra-live-es.json

Add --dry-run to any stage to skip external calls and write placeholder files.

Adding a New Job

Create a JSON file in nexetra-media/jobs/:

{
  "job_id": "product-demo-fr",
  "title": "Nexetra Product Demo",
  "duration_seconds": 60,
  "languages": ["en", "fr", "de"],
  "formats": ["16:9", "9:16", "1:1"],
  "style": "slides",
  "cta": "Book a demo at nexetra.com",
  "status": "draft"
}

The dashboard shows the new job card within 4 seconds (next SSE cycle). Run it with:

python pipeline\run_batch_pool.py --job jobs\product-demo-fr.json

Managing the Dashboard Service

The viewer runs as a systemd user service on ubuntu-1 (10.0.0.200).

Check status (SSH)

ssh user@10.0.0.200
systemctl --user status nexetra-viewer

View live logs

journalctl --user -u nexetra-viewer -f

Redeploy after code changes

# From nexetra-media/ on your local machine
python viewer\deploy.py --host ubuntu-1

Restart without re-syncing code

python viewer\deploy.py --host ubuntu-1 --restart-only

Stop / Start manually

systemctl --user stop nexetra-viewer
systemctl --user start nexetra-viewer
The service is enabled at user login and auto-restarts after crashes with a 5-second back-off.

Pool Configuration

Edit nexetra-media/config.yaml:

compute_pool:
  deny_hosts:           # Never touched — mission-critical hosts
    - "linux-1"
    - "linux-2"
    - "das-Mac-mini.local"

  allow_name_patterns:  # Only these patterns are eligible
    - "^gx10"
    - "^Lab-"
    - "^ubuntu-"

  remote_root_candidates:  # Where to find nexetra-media on each remote host
    - "/home/gx10/nexetra-media"
    - "/home/user/nexetra-media"
    - "~/nexetra-media"

Provisioning New Remote Hosts

  1. Ensure the host appears in workstations.csv (name, IP, username)
  2. Add its password to secrets.local.md in the standard block format
  3. Deploy the runtime:
    python viewer\deploy.py --host ubuntu-3

The host will appear as Remote✓ in the dashboard within 4 seconds of the next health check.

"⚠ Reconnecting…" in header SSE connection dropped Browser auto-reconnects; check service logs if persistent
